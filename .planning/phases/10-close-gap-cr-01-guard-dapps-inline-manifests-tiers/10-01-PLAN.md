<objective>
Close the v1.1 milestone-audit CR-01 gap: extend ROB-05's registry-only array-shape guard to
`loadManifests()`'s other two tiers (`dapps` and inline `manifests`) via a shared closure-local
`coerceManifestArray()` helper, so an untyped IIFE/static-HTML consumer that passes a wrong-shape
`dapps`/`manifests` config gets a visible `dx:error` and a safe empty manifest list instead of an
uncaught `TypeError` before `window.__DXKIT__` is exposed.

Purpose: DxKit's first-class deployment target is untyped static/IIFE HTML — TypeScript catches a
wrong-shape config only for typed bundler consumers. ROB-05 hardened the terminal registry tier;
the two non-terminal config tiers still trust the raw value's shape and crash `init()` before the
public API bridge is installed. This satisfies ROB-06 and the project's "never silent" core value.

Output: `coerceManifestArray()` helper in `src/shell.ts`, a restructured `loadManifests()`, and a
regression suite in `tests/shell.test.ts` covering all wrong-shape/empty/fail-closed cases.
</objective>

<locked_decisions>
These were delegated to the research-recommended defaults (discuss-phase skipped) and are NON-NEGOTIABLE:

- **A1 — fail closed.** A present-but-wrong-shape `dapps`/`manifests` value FAILS CLOSED:
  `coerceManifestArray()` emits `dx:error` and the tier returns `[]` immediately. It MUST NOT
  silently fall through to the next configured tier. The fail-closed vs. fall-through decision is
  made by branching on `coerced === null` — never on `.length` (which conflates the `null` failure
  case with a valid empty array). Rationale: falling through would silently swap manifest sources,
  violating the "never silent" core value.
- **A2 — ROB-06.** The new requirement ID is `ROB-06` (registered at plan-authoring time). Do not
  reuse the closed `ROB-05` or the audit's internal `CR-01` label (which collides with an unrelated
  Phase-6 decision-log usage).
