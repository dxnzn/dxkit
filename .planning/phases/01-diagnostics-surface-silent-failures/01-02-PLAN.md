<objective>
Make localStorage read/write failures visible in the wallet, theme, and settings plugins
(DIAG-02). Today each plugin swallows storage errors in bare `catch {}` blocks, so a
developer whose settings/theme/wallet-selection silently stop persisting has no signal.

This plan converts each swallowing catch into a `dx:error` emit that names the plugin and
operation via source `plugin:<name>:storage:<op>` (op ∈ {read, write}), while keeping the
"storage entirely unavailable" path silent (D-07) and preserving defaults-fallback on a
corrupted restore (D-08).

Purpose: `dx:error` becomes a trustworthy signal for genuine storage failures without
flooding on expected environment conditions (SSR, private mode).

Output: Three modified plugin index.ts files plus one regression test per plugin.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| plugin storage code → `dx:error` listeners | Wrapped error messages emitted on the bus are visible to every `dx:error` handler on the origin. |
| localStorage (browser/user-controlled) → plugin restore path | `getItem`/`JSON.parse` consume attacker- or corruption-controlled bytes on restore(). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01B-01 | Information Disclosure | `plugin:<name>:storage:<op>` error messages | low | mitigate | Wrapped Error messages carry only the operation identity and the caught exception's `err.message` (e.g. QuotaExceededError) — NOT the stored settings values, wallet provider id, or the parsed JSON payload. Reviewers must confirm no storage value/payload is interpolated into the emitted message. |
| T-01B-02 | Tampering / DoS | restore() consuming corrupted localStorage JSON | low | mitigate | `catch (err)` emits `:read` and falls back to defaults (D-08) — a hostile/corrupt value cannot crash init or block the plugin; parsing is bounded by localStorage size limits. |
| T-01B-03 | Denial of Service | repeated identical storage-failure emits | low | accept | No dedupe (D-09): a persistently-full quota emits per write. Each emit is a bounded synchronous CustomEvent; the flood is itself the intended signal. Throttle/dedupe explicitly rejected and deferred. |

No high-severity threats on this path (block-on threshold is `high`). No package installs
in this phase (zero-dep posture), so no supply-chain/legitimacy checkpoint applies.
</threat_model>
