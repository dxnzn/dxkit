<objective>
Close verification gap Truth #4 (04-VERIFICATION.md, re-opened as CR-01 in the fresh 04-REVIEW.md): the D-01 "last-navigation-wins" invariant is still violated by a distinct interleaving that gap-closure plan 04-04 did not cover. `mountDapp`'s `finally` block (src/shell.ts:412-414) unconditionally runs `pendingMountId = null`, even when the settling call is stale/superseded by a newer in-flight mount for a DIFFERENT dapp. That clobbers the newer mount's marker, so `handleRouteChange`'s null-manifest branch (`if (pendingMountId) lifecycle.invalidatePendingMount(pendingMountId)`, src/shell.ts:364 — the 04-04 fix) becomes a no-op exactly when it is needed.

Reproduced interleaving: navigate('/a') suspends at the entry gate (pendingMountId='a') -> navigate('/b') supersedes A (pendingMountId='b') -> A's held loader releases; A is stale (isStale aborts it) but its finally clears pendingMountId -> navigate('/nowhere') finds pendingMountId already null and skips invalidation -> B's loader releases and commits dx:mount/dx:dapp:mounted while the route is '/nowhere'.

Purpose: Restore the D-01 invariant for the A->B-overlap-then-unmatched-route interleaving by (1) fixing the root-cause corruption of the shell-level pendingMountId slot and (2) decoupling handleRouteChange's null-branch invalidation from that shell-level slot entirely, so it reads the lifecycle's own ground-truth (inFlightMountId). Both together close the class, not just the one repro.
Output: A guarded finally in src/shell.ts, a new `invalidateAnyPendingMount()` entry point in src/lifecycle.ts wired into handleRouteChange's null branch, and a committed stress scenario that fails before the fix and passes after.

Scope discipline: Close Truth #4 (CR-01 reopened) ONLY. ## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dapp script -> shell container (#dx-mount) | A dapp's template/entry is injected into the shared mount container; only the dapp matching the current route may own it |
| browser navigation -> shell route resolution | User-driven URL changes (including to unmatched routes) drive mount/unmount decisions; overlapping navigations must not let a superseded mount commit |

## STRIDE Threat Register
(ASVS level 1; blocking threshold: high)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Spoofing | handleRouteChange null branch (src/shell.ts) | high | mitigate | This plan's fix: an overlapping A->B race can currently clobber pendingMountId so a stale-slot bypass lets dapp B commit its DOM/events under an unmatched URL, misrepresenting which dapp owns the container. Reading lifecycle's own inFlightMountId via invalidateAnyPendingMount() prevents the stale commit regardless of shell bookkeeping — closing the misattribution. |
| T-04-02 | Tampering | #dx-mount container | medium | mitigate | A superseded mount injecting/owning content in a container the shell has navigated away from — the generation bump before every DOM write/commit gate prevents the stale write. |
| T-04-03 | Tampering | shell-level pendingMountId slot | medium | mitigate | A stale call's unconditional finally erases a newer, unrelated mount's in-flight marker, corrupting shared state other guards read. The guarded finally (clear only when pendingMountId === manifest.id) prevents cross-mount state corruption. |

No dapp-secret exposure — event payloads carry route/id/container only. Accepted residual: no mitigation beyond the fix. |
