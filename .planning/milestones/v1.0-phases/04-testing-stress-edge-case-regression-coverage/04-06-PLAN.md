The bug: `mountDapp`'s same-dapp dedupe (`src/shell.ts:392`, `if (pendingMountId === manifest.id) return;`) checks only id equality, never whether the in-flight call it dedupes against was already invalidated by `lifecycle.invalidateAnyPendingMount()` / `invalidatePendingMount()`. When `handleRouteChange`'s null branch invalidates an in-flight mount for A (`src/shell.ts:367`) it bumps the lifecycle's `mountGeneration` but leaves the shell-level `pendingMountId` parked at `'a'`. A subsequent `navigate('/a')` then hits the dedupe and is SILENTLY DROPPED — the invalidated first mount later fails `isStale()`, and its guarded `finally` clears `pendingMountId`. Final state: `getCurrentRoute()==='/a'` but nothing mounted, no `dx:mount`, no error. The drop window lasts as long as the invalidated mount's slowest loader (up to the 30s default timeout). `disableDapp` shares the identical hole at its own invalidation call site (`src/shell.ts:136`).

Reproduced interleaving: `navigate('/a')` suspends at a held loader gate (pendingMountId='a', lifecycle in-flight for 'a') -> `navigate('/nowhere')` drives `handleRouteChange(null)` -> invalidateAnyPendingMount() bumps mountGeneration but pendingMountId stays 'a' -> `navigate('/a')` again -> `mountDapp('a')` early-returns at the same-id dedupe -> the re-navigation is lost.

Purpose: Make the shell dedupe slot call-scoped and liveness-aware so a re-navigation to a dapp whose pending mount was invalidated starts a FRESH mount, while the legitimate same-dapp dedupe (two genuinely-concurrent identical mounts) and the 04-04/04-05 fixes are preserved. Per CONTEXT.md's standing policy (D-04/D-05): correctness bugs surfaced by the new tests are fixed in-phase with their tests; the suite lands green (no test.fails).
Output: A call-scoped `pendingMountToken` + a `releasePendingMount()` helper in src/shell.ts wired into both invalidation call sites, plus two committed, green stress regression scenarios (the WR-11 re-navigation test and its disableDapp twin) that fail if the fix is reverted.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser navigation -> shell route resolution | User-driven URL changes (including to unmatched routes and disable/enable transitions) drive mount/unmount decisions; a re-navigation must not be silently dropped, and no superseded mount may commit |
| dapp script/template -> shell container (#dx-mount) | Only the dapp matching the most recent navigation may own the shared mount container; a stale/invalidated mount must never inject or commit |

## STRIDE Threat Register
(ASVS level 1; blocking threshold: high)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Denial of Service | mountDapp same-id dedupe (src/shell.ts:392) + shell pendingMountId slot | high | mitigate | This plan's fix: an invalidated in-flight mount leaves pendingMountId parked, so the same-id dedupe silently drops every re-navigation to that dapp for up to the 30s loader timeout — a self-inflicted availability hole where an explicit user navigation renders nothing. Making the slot call-scoped (pendingMountToken) and freeing it on invalidation (releasePendingMount at both call sites) restores the re-navigation, closing the drop window. |
| T-04-02 | Spoofing | dx:route:changed vs actual mounted dapp | medium | mitigate | Before the fix, getCurrentRoute()==='/a' while nothing is mounted and getCurrentDapp() is null — the route stream misrepresents which dapp owns the container. The fix makes the DOM/dx:mount state agree with the last navigation. |
| T-04-03 | Tampering | shell pendingMountId slot | medium | mitigate | A stale/invalidated call's finally clearing a newer, still-in-flight mount's slot (WR-01) corrupts shared dedupe state other guards read. The call-scoped token (finally clears only when pendingMountToken === myToken) prevents cross-call slot corruption. |
| T-04-04 | Information Disclosure | dx:mount / dx:dapp:mounted payloads | low | accept | Event payloads carry route/id/container only — no dapp secrets. No mitigation needed beyond the correctness fix; accepted residual. |
