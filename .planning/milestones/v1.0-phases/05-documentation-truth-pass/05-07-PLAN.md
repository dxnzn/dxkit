## Trust Boundaries

| docs/security.md → consumer security posture | Consumers configure CSP and sanitizers from this doc; wrong guidance directly weakens (or falsely reassures) their deployment |
| template/manifest source → shell (innerHTML) | Untrusted template HTML crosses into the DOM via innerHTML; sanitizer + CSP are the documented mitigations |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-07a | Tampering | CSP policy examples | high | mitigate | A CSP example that blocks DxKit's own dynamic <script>/<link> injection would lead consumers to disable CSP entirely; every directive reasoned against src/lifecycle.ts loaders and verified not to block required injection. |
| T-05-07b | Repudiation | meta-tag CSP directive support | high | mitigate | Presenting header-only directives (frame-ancestors/report-uri/sandbox) in a <meta> example silently drops enforcement; doc explicitly states the meta-safe directive list and the non-meta directives (Pitfall 4). |
| T-05-07c | Spoofing (false assurance) | DOMPurify sanitizer recipe | high | mitigate | A recipe implying entry scripts are sanitized gives false assurance; doc states the sanitizer covers template HTML only, entry scripts are trusted code (Phase 3 D-14). |
| T-05-07d | Information Disclosure | limitations inventory | medium | mitigate | Omitting the localStorage-plaintext or theme/settings storageKey-collision facts would let consumers persist secrets or assume isolation that isn't there; both documented explicitly (D-11). |

No package installs — DOMPurify is a bring-your-own consumer choice referenced in prose only (version 3.4.12 spot-checked in RESEARCH); no `T-05-SC` checkpoint required.
