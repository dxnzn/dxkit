<objective>
Add an optional per-fetch load timeout to the lifecycle manager (ROB-01) so a hung
script/style/template URL can no longer freeze a mount indefinitely. The default is a
30-second timeout that ships enabled (a deliberate breaking change, D-02), with `timeout: 0`
or `Infinity` as the documented opt-out that restores today's hang-forever behavior (D-03).

Purpose: mounts become un-hangable by default — the point of ROB-01 — while custom-loader
consumers keep a graceful degradation path.
Output: `src/lifecycle.ts` gains a `timeout` option and true-abort machinery; `tests/lifecycle.test.ts`
gains fake-timer coverage for every timeout path.
</objective>

  <action>
    Add optional `timeout?: number` to `LifecycleManagerOptions` (JSDoc: milliseconds per fetch; default 30000; `0` or `Infinity` disables — the IPFS-gateway escape hatch per D-03). In `createLifecycleManager`, resolve `const timeoutMs = options.timeout ?? 30000;` and treat `timeoutMs <= 0 || timeoutMs === Infinity` as "no timeout" everywhere below.

    Built-in loaders self-abort (D-06). Parameterize the three default-loader factories by `timeoutMs`:
    - `defaultScriptLoader` / `defaultStyleLoader`: when timeout is active, start a `setTimeout(timeoutMs)` after appending the node; on fire, remove the injected `script`/`link` node from `document.head`, null its `onload` and `onerror`, and `reject` with a timeout Error. On normal `onload`/`onerror`, clear the timer. This guarantees a late-arriving asset cannot execute into an already-cleared or next dapp.
    - `defaultTemplateLoader`: create an `AbortController`, pass `signal` to `fetch(src, { signal })`, start a `setTimeout(timeoutMs)` that calls `controller.abort()`; map the resulting abort rejection to a timeout Error; clear the timer on success.

    Custom loaders stay opaque (D-07): the loader type signatures do NOT change (no `AbortSignal` param). Add an internal helper (e.g. `withTimeout(loader, timeoutMs)`) that returns a wrapped loader doing `Promise.race` between the loader's promise and a timeout rejection — the underlying load continues in the background (a documented degradation; note this in a one-line why-comment). At the seam (:80-84), when the consumer supplied `options.scriptLoader`/`styleLoader`/`templateLoader`, wrap it with `withTimeout`; otherwise use the timeout-parameterized default loader. When timeout is disabled, use loaders as-is (no race, no setTimeout) so hang-forever is restored exactly.

    Timeout Error message must name the timeout ms and the URL, in the style of the existing `Failed to load dapp template: ${src} (${res.status})` message (e.g. a "Timed out loading dapp <asset> after <ms>ms: <src>" shape). Reuse the existing per-asset `dx:error` source strings — DO NOT add or change source strings; the existing mount-sequence try/catch blocks (styles :106-115 non-blocking, template :118-129 abort-return, dependency :131-146 clear+return, entry :148-159 clear+return) already emit with the correct source and already honor the blocking/non-blocking split (D-04/D-05). A timeout-rejecting loader flows through them unchanged — do not duplicate emit logic in the mount body.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| remote asset URL → shell DOM | Untrusted script/style/template content crosses into `document.head` / the mount container. A hung or slow URL is a denial-of-availability vector; a late-firing node is a stale-execution vector. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Denial of Service | mount() awaiting a hung loader | high | mitigate | Per-fetch 30s default timeout aborts the load and emits dx:error instead of hanging the mount (D-02). |
| T-02-02 | Tampering / Elevation | built-in script/style node firing after timeout | high | mitigate | On timeout, remove the injected node and null onload/onerror so a late asset cannot execute into an already-cleared or next dapp (D-06). |
| T-02-03 | Denial of Service | custom opaque loader that never resolves | medium | mitigate | Promise.race hang guard abandons the wait and emits dx:error; underlying load continues in background (documented degradation, D-07). |
| T-02-04 | Availability (intentional opt-out) | timeout: 0 / Infinity restoring hang-forever | low | accept | Documented escape hatch for legitimately-slow IPFS gateways; consumer opts in explicitly (D-03). |

No package-manager installs in this plan (zero runtime deps posture); no supply-chain checkpoint required.
</threat_model>
