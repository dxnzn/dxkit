<objective>
Cache dapp templates by URL (ROB-03) so repeated mounts of the same dapp reuse the fetched
HTML instead of re-fetching it, with an explicit invalidation path. Caching is on by default
(D-09, safe for the content-addressed/immutable IPFS/static target) with `cacheTemplates: false`
to opt out for dev/live-editing. The cache is a per-manager `Map<url, html>` in the
`createLifecycleManager` closure (no module-level singleton, D-11), and two new interface
methods — `clearTemplateCache()` and `invalidateTemplate(url)` — give full-reset and targeted
granularity (D-10).

Purpose: eliminate redundant template fetches on re-mount while keeping a clean invalidation
story and per-instance test isolation.
Output: `src/lifecycle.ts` gains the cache + options + methods; `tests/lifecycle.test.ts` gains
cache coverage.

Note (sequencing): this plan builds on 02-01's timeout-wrapped `loadTemplate` in the same file.
The cache wraps OUTERMOST — check the cache first; on a miss, call the (already timeout-wrapped)
loader; cache only on success. A cache hit returns immediately and never touches the fetch or its
timeout (D-12).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cached template HTML → mount container innerHTML | Cached HTML is re-injected via innerHTML on re-mount. The cache stores only what a successful fetch returned for a given URL. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-08 | Tampering | stale/poisoned template served from cache | low | mitigate | Only successfully-fetched HTML is cached, keyed by exact URL; an explicit invalidation API (clearTemplateCache / invalidateTemplate) and the cacheTemplates:false opt-out cover dev/live-editing and forced-refresh needs (D-10/D-12). Content-addressed IPFS/static target makes URL-keyed cache safe. |
| T-02-09 | Information Disclosure | template cache surviving across manager teardown | low | accept | Cache is per-manager closure state, garbage-collected with the manager instance; no cross-instance or module-level leakage (D-11). |

Template `innerHTML` sanitization is explicitly out of scope here — that is SEC-01 (Phase 3). This plan does not change the injection surface; it only avoids re-fetching identical HTML.

No package installs; no runtime dependency added.
